## 二、详细发言稿

### ▶ 1. 项目背景与目标（8 分钟）

各位组员，欢迎参加 LuminFlow 项目启动会议。

#### 1.1 行业痛点

我们本次黑客松的切入点是**审计行业**。传统审计面临三个核心痛点：

- **效率瓶颈**：审计师在风险评估、抽样决策、控制测试等环节依赖大量手工操作和 Excel，一个中型审计项目从计划到出具报告通常需要 4–6 周。
- **沟通断层**：审计发现需要向 CFO、内审委员会、IT 部门等不同受众汇报，但技术语言与商业语言之间存在巨大鸿沟——技术人员讲"控制缺陷"，CFO 关心"财务影响多少钱"。
- **知识孤岛**：审计经验沉淀在资深合伙人大脑中，初级审计师缺乏实时指导，质量复核依赖事后抽检而非事中预警。

#### 1.2 我们的解决方案

LuminFlow 是一个**基于 COSO 2013 框架的智能审计 AI 助手平台**，核心理念是——**让 AI 成为每一位审计师的实时搭档**。

我们的差异化定位：

- 不是通用 ChatBot，而是**深度嵌入审计工作流**的专业 AI
- 基于 COSO 2013（17 项原则、5 大要素）的领域知识驱动
- 多角色视角（CFO / 审计师 / 合伙人）适配不同决策场景
- 四管道 AI 引擎协同：风险分析 → 抽样推荐 → 影响模拟 → 叙事生成

#### 1.3 演示目标

到第三天 Demo 时，我们将展示一个完整的审计工作闭环：从项目仪表板监控 → 审计目标管理 → AI 抽样决策 → 控制缺陷影响模拟 → 面向 CFO 的叙述生成 → 团队协作发布。每一个环节都有 AI 深度参与。

### ▶ 2. 技术栈说明（7 分钟）

#### 2.1 整体架构

```
┌─────────────────────────────────────────────────┐
│                   前端 (Port 5173)               │
│  React 19 + TypeScript + Vite 8 + TailwindCSS   │
│  Zustand (状态) + TanStack Query (数据)          │
│  D3.js / ECharts / Recharts (可视化)            │
│  Framer Motion (动效)                            │
└──────────────────┬──────────────────────────────┘
                   │ HTTP / SSE Streaming
┌──────────────────▼──────────────────────────────┐
│                 后端 (Port 8000)                  │
│  Python 3.12 + FastAPI + Pydantic v2             │
│  httpx → DeepSeek API (流式)                     │
│  YAML Prompt 模板引擎                            │
│  4 大 AI 引擎 Pipeline                           │
│  ① Risk Analyzer   — 风险分析                    │
│  ② Sample Recommender — 抽样推荐                 │
│  ③ Impact Simulator — 影响模拟                   │
│  ④ Narrative Generator — 叙述生成               │
└─────────────────────────────────────────────────┘
```

#### 2.2 前端技术选型理由

| 技术 | 理由 |
|------|------|
| **React 19 + Vite 8** | 生态成熟、HMR 极快、TS 原生支持 |
| **TailwindCSS** | 原子化 CSS，组件样式零冲突，适合快速迭代 |
| **Zustand** | 轻量状态管理，比 Redux 减少 80% 样板代码 |
| **TanStack Query** | 服务端状态缓存、自动重取、乐观更新 |
| **D3.js + ECharts** | D3 做力导向图/审计树等定制可视化，ECharts 做标准图表 |
| **SSE 流式** | Chat 模块使用 Server-Sent Events，打字机效果 |

#### 2.3 后端技术选型理由

| 技术 | 理由 |
|------|------|
| **FastAPI** | 异步高性能、自动 OpenAPI 文档、Pydantic 类型安全 |
| **DeepSeek API** | 高性价比大模型，支持流式输出，中文能力强 |
| **YAML Prompt 模板** | Prompt 与代码分离，非技术人员可直接调优 |
| **Mock Fallback** | 无 API Key 时自动降级为 Mock 响应，确保 Demo 可运行 |

#### 2.4 关键依赖

- **前端核心**：`react@19`、`react-router-dom@7`、`zustand@5`、`@tanstack/react-query@5`、`d3@7`、`echarts@6`、`framer-motion@12`、`lucide-react`、`axios`
- **后端核心**：`fastapi`、`uvicorn`、`httpx`、`pydantic`、`pyyaml`、`python-dotenv`

### ▶ 3. 核心功能模块介绍（15 分钟）

基于现有代码结构，我们有 **6 个核心页面 + 1 个全局 AI 助手**：

#### 3.1 审计门户 Dashboard（`DashboardPage`）

**代码位置**：`src/pages/DashboardPage.tsx` + `src/components/dashboard/`

**功能**：
- **KPI 卡片**（`KPICards`）：展示 4 个核心指标（完成率、发现率、逾期率、客户满意度），含目标进度条和趋势图标
- **角色自适应视图**：CFO 看战略总览 → 审计师看执行细节 → 合伙人看质量复核
- **风险热力图**（`RiskHeatmap`）：RCM 矩阵二维可视化，COSO 五要素 × 控制类型交叉
- **状态环形图**（`StatusDoughnut`）：控制测试完成/进行中/待办分布
- **审计步骤条**（`AuditStepper`）、待办清单（`PendingItems`）、动态时间线（`ActivityTimeline`）

**核心逻辑**：
- 通过 `useAuthStore` 获取当前角色，驱动不同布局策略
- `getKPIMetrics(currentRole)` 返回角色相关指标（数据来自 `src/data/kpi-metrics.json`）
- 加载态使用 `Skeleton` 组件保证体验连贯

---

#### 3.2 AI 影响模拟器 Impact（`ImpactPage`）

**代码位置**：`src/pages/ImpactPage.tsx` + `server/engines/impact_simulator.py` + `server/routes/impact.py`

**功能**：
- **事件配置面板**（`EventConfig`）：选择事件类型、严重程度（1–10）、影响维度（COSO 五要素）、应对策略
- **力导向图**（`ForceGraphD3`）：D3.js 力仿真，节点大小/颜色映射影响分数，边权重表示传播强度
- **前后对比表**（`ComparisonTable`）：各维度影响值 Before/After/Delta
- **AI 建议卡片**（`AIRecommendations`）：基于模拟结果生成缓解建议

**后端接口**：`POST /api/impact/simulate`，返回 `nodes + links + comparison + recommendations + summary`

---

#### 3.3 审计目标管理 Objective（`ObjectivePage`）

**代码位置**：`src/pages/ObjectivePage.tsx` + `src/components/objective/`

**功能**：
- **审计树可视化**（`AuditTreeD3`）：四层树形结构（根目标 → 风险域 → 控制 → 发现），可搜索、缩放、点击展开详情
- **雷达图**（`RadarChart`）：COSO 五要素覆盖度评分
- **双进度条**（`DualProgress`）：设计有效性 vs 执行有效性对比
- **发现表格**（`FindingsTable`）：按严重程度排序的审计发现清单
- **节点详情抽屉**（`NodeDetail`）：点击任意节点查看详情
- **AI 洞察面板**（`AIInsightPanel`）：基于当前树状态的 AI 分析建议

**数据结构**：`AuditObjective → RiskArea → Control → Finding` 四级嵌套，参考 `src/types/audit.ts`

---

#### 3.4 智能抽样 Sampling（`SamplingPage`）

**代码位置**：`src/pages/SamplingPage.tsx` + `server/engines/sample_recommender.py` + `server/routes/sampling.py`

**功能**：
- **样本量卡片**（`SampleSizeCard`）：推荐样本量范围（min–max），含置信度
- **时间窗口卡片**（`TimeWindowCard`）：审计覆盖时间段
- **参数调节器**（`ParamAdjuster`）：风险阈值、置信水平、提升因子实时调节
- **旭日图**（`SunburstChart`）：文档类型分布层级可视化
- **AI 逻辑解释器**（`AILogicExplainer`）：解释抽样逻辑（为什么上浮、为什么某控制需更多样本）
- **历史趋势图**（`HistoryBarChart`）+ **样本详情表**（`SampleTable`）

**后端接口**：`POST /api/sampling/preview`，基于 RAWTC 评分、上年度缺陷、风险阈值自动计算

---

#### 3.5 团队协作 Team Panel（`TeamPanelPage`）

**代码位置**：`src/pages/TeamPanelPage.tsx` + `src/components/team/`

**功能**：
- **可见性管理**（`VisibilityPanel`）：控制审计发现对不同角色的可见度
- **AI 审批队列**（`ApprovalQueue`）：合伙人审批工作流
- **客户互动图表**（`InteractionChart`）：PBC 提交、反馈统计
- **热门问题栏**（`HotQuestionsBar`）：客户高频咨询聚合
- **证据检查器**（`EvidenceChecker`）：AI 辅助证据完整性校验
- **发布对话框**（`PublishDialog`）：报告发布与通知

---

#### 3.6 AI 对话助手（全局）

**代码位置**：`src/components/ai/AIChatDrawer.tsx` + `server/routes/chat.py`

**功能**：
- 右侧滑出式对话面板，全局可用
- SSE 流式响应（打字机效果）
- 角色感知（自动注入当前角色和页面上下文）
- 快捷提示词（`QuickPrompts`）：根据当前页面推荐问题
- 对话历史管理（清空/保留）

**后端**：`POST /api/chat`，支持 DeepSeek API 流式 / Mock 降级

---

### ▶ 4. 开发工作流与协作机制（10 分钟）

#### 4.1 分支策略

```
main ─────●──────────●──────────●── 最终演示版本
           \         /          /
feature/dashboard    /          /
feature/sampling ────          /
feature/impact ────────────────
```

- `main` 分支：稳定可演示版本，只接受 PR
- 每人从 `main` 拉取自己的 feature 分支
- **禁止**直接 push 到 main

#### 4.2 前后端协作协议

**接口先行**：后端同学先定义好 API Schema（Pydantic models），前端同学用 TypeScript 类型同步。

**示例**：
```python
# 后端 server/models/schemas.py
class SamplingPlanResult(BaseModel):
    sample_size_range: SampleSizeRange
    confidence_level: float
    document_distribution: list[DocumentDistribution]
    logic_explanation: list[str]
    elevated_controls: list[ElevatedControl]
```

```typescript
// 前端同步类型（在 src/types/sampling.ts）
export interface SamplingPlanResult {
  sampleSizeRange: { min: number; max: number }
  confidenceLevel: number
  documentDistribution: DocumentDistribution[]
  logicExplanation: string[]
  elevatedControls: ElevatedControl[]
}
```

#### 4.3 本地开发环境

```bash
# 终端 1：启动后端
cd server
pip install -r requirements.txt
uvicorn main:app --reload --port 8000

# 终端 2：启动前端
npm run dev    # → http://localhost:5173
```

**API 文档**：后端启动后访问 `http://localhost:8000/docs` 查看 Swagger UI

### ▶ 6. 潜在挑战与应对策略（7 分钟）

#### 6.1 技术风险

| 风险 | 影响 | 概率 | 应对策略 |
|------|------|------|----------|
| **DeepSeek API 不稳定/限流** | AI 功能不可用 | 中 | 后端已内置 Mock Fallback，Chat/Sampling/Impact 均有本地降级逻辑 |
| **D3.js 可视化复杂度过高** | 力导向图/审计树无法如期完成 | 中 | 已拆分为独立组件，可先展示静态版本再添加交互 |
| **前后端接口不一致** | 联调阻塞 | 高 | Schema First：接口变更必须同步更新 Pydantic + TS 类型 |
| **TypeScript 类型报错** | 编译失败 | 低 | 使用 `tsc --noEmit` 预检，IDE 实时提示 |
| **时间不足** | 部分功能砍掉 | 高 | 已按优先级排列：Dashboard > AI Chat > Sampling > Impact > Objective > Team Panel，后两个可降级为静态展示 |

#### 6.2 协作风险

| 风险 | 应对策略 |
|------|----------|
| 成员对技术栈不熟悉 | 第一天上午集中配环境，老手帮新手 |
| 任务分配不均 | 会议第 7 环节现场确认，每人认领明确模块 |
| 夜间疲劳导致 Bug | 每天 23:00 后禁止合并代码，避免"深夜炸弹" |

#### 6.3 Demo 风险

| 风险 | 应对策略 |
